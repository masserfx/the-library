# Initialize Mimoni in a Project

## Context

Set up mimoni orchestration in a new or existing project. This creates a `mimoni.yaml` configuration file tailored to the project's tech stack by auto-detecting the project type from existing files. It also ensures the `.mimoni/` state directory is gitignored so local run state is not committed.

This is typically the first mimoni command a user runs in a project, before using `/library mimoni run <task>`.

## Steps

### 1. Check Prerequisites

Verify the current directory is a git repository:

```bash
git rev-parse --git-dir
```

- **If it succeeds** → this is a valid git repo. Proceed to the next check.
- **If it fails** → stop and tell the user:
  > `Not a git repository. Initialize one with git init before running mimoni init.`

Check if `mimoni.yaml` already exists in the project root:

```bash
test -f mimoni.yaml
```

- **If `mimoni.yaml` already exists** → warn the user and skip creation:
  > `mimoni.yaml already exists in this project. To reinitialize, delete it first and re-run /library mimoni init.`
  >
  > Stop here — do not overwrite the existing configuration.
- **If `mimoni.yaml` does not exist** → proceed to step 2.

### 2. Create mimoni.yaml with Project Defaults

Create a `mimoni.yaml` configuration file by copying the canonical template from The Library and adjusting the validation commands based on the detected project type. The canonical template lives at `<LIBRARY_SKILL_DIR>/mimoni.example.yaml` — always copy from there to prevent template drift.

**Copy the canonical template:**

```bash
cp <LIBRARY_SKILL_DIR>/mimoni.example.yaml mimoni.yaml
```

**Auto-detect the project type** by checking for known project files:

```bash
# Check for project type indicators (in priority order)
test -f package.json      # Node.js / JavaScript / TypeScript
test -f Cargo.toml        # Rust
test -f setup.py          # Python (setup.py)
test -f pyproject.toml    # Python (modern)
test -f requirements.txt  # Python (pip)
test -f go.mod            # Go
test -f Gemfile           # Ruby
test -f pom.xml           # Java (Maven)
test -f build.gradle      # Java / Kotlin (Gradle)
test -f composer.json      # PHP
test -f Makefile          # Generic (fallback)
```

**Set validation commands based on detected project type:**

| Detected File       | `validation.lint`        | `validation.typecheck`        | `validation.test`          |
| -------------------- | ------------------------ | ----------------------------- | -------------------------- |
| `package.json`       | `npm run lint`           | `npm run typecheck`           | `npm test`                 |
| `Cargo.toml`         | `cargo clippy`           | `cargo check`                 | `cargo test`               |
| `setup.py` / `pyproject.toml` / `requirements.txt` | `ruff check .`           | `mypy .`                      | `pytest`                   |
| `go.mod`             | `golangci-lint run`      | `go vet ./...`                | `go test ./...`            |
| `Gemfile`            | `rubocop`                | `echo "no typecheck"`         | `bundle exec rspec`        |
| `pom.xml`            | `mvn checkstyle:check`   | `mvn compile`                 | `mvn test`                 |
| `build.gradle`       | `./gradlew lint`         | `./gradlew compileJava`       | `./gradlew test`           |
| `composer.json`      | `composer run lint`      | `composer run analyze`        | `composer run test`        |
| `Makefile` (fallback)| `make lint`              | `make typecheck`              | `make test`                |
| None detected        | `echo "no lint configured"`| `echo "no typecheck configured"` | `echo "no test configured"` |

If multiple project files are found, use the **first match** in the priority order listed above.

**For Node.js projects**, refine further by inspecting `package.json` scripts:

```bash
python3 -c "
import json
pkg = json.load(open('package.json'))
scripts = pkg.get('scripts', {})
print('lint' if 'lint' in scripts else 'no-lint')
print('typecheck' if 'typecheck' in scripts else 'no-typecheck')
print('test' if 'test' in scripts else 'no-test')
"
```

- If a script is missing from `package.json`, use `echo "no <command> configured"` as a placeholder so the user knows to fill it in.

**Modify the copied `mimoni.yaml`** — update only the `validation` section with the detected commands. Replace the default `npm run lint`, `npm run typecheck`, and `npm test` values with the commands from the auto-detection table above. Also update the header comment to indicate it was generated:

```bash
# Update the first comment line to indicate generation
sed -i '' '1s/.*/# mimoni.yaml — Configuration for mimoni agent orchestration./' mimoni.yaml
sed -i '' '2s/.*/# Generated by \/library mimoni init. Customize for your project./' mimoni.yaml
```

All other sections (pr, agent, context) retain the canonical defaults from `mimoni.example.yaml`. The user can customize them after initialization.

### 3. Add .mimoni/ to .gitignore

The `.mimoni/` directory stores local run state and should not be committed to the repository.

**Check if `.gitignore` exists and already contains `.mimoni/`:**

```bash
grep -qF '.mimoni/' .gitignore 2>/dev/null
```

- **If `.gitignore` exists and already contains `.mimoni/`** → skip this step.
- **If `.gitignore` exists but does not contain `.mimoni/`** → append it:

```bash
echo "" >> .gitignore
echo "# Mimoni run state (local, not committed)" >> .gitignore
echo ".mimoni/" >> .gitignore
```

- **If `.gitignore` does not exist** → create it with the entry:

```bash
echo "# Mimoni run state (local, not committed)" > .gitignore
echo ".mimoni/" >> .gitignore
```

### 4. Confirmation

Tell the user what was created and suggest the next step. Adjust the `.gitignore` line based on what happened in Step 3:

- **If `.gitignore` was modified** (entry was appended or file was created):

> ✅ **Mimoni initialized.**
> - Created `mimoni.yaml` with **<detected project type>** defaults.
> - Added `.mimoni/` to `.gitignore`.

- **If `.mimoni/` was already in `.gitignore`** (Step 3 was skipped):

> ✅ **Mimoni initialized.**
> - Created `mimoni.yaml` with **<detected project type>** defaults.
> - `.mimoni/` already in `.gitignore`.

In both cases, include the next steps:

> **Next steps:**
> 1. Review `mimoni.yaml` and adjust validation commands if needed.
> 2. Run `/library mimoni run "<task description>"` to start your first mimoni orchestration.
> 3. Run `/library mimoni list` to see available blueprint workflows.
